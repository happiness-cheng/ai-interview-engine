# 评分与复习规则

## 全局前提：未教学内容不评分

评分只适用于以下内容：

- 已经正式教学过的知识
- tracker 显示已经学习过的知识
- 已确认属于用户前置知识的内容

首次引出的新知识不评分。用户回答"不知道"时，切换为微型教学，而不是判 Again。

区分：

```
已学过但现在完全不会 → Again
首次接触而不知道 → 尚未教学，不评分
```

---

## 四级评分

### Again

满足任意情况：

- 完全不会
- 核心结论错误
- 主要知识已经忘记
- 存在根本性误解
- 把相反概念混淆
- 讲解后仍不能完成验证

忘记答案必须判为 Again，不能因为"好像有印象"判为 Hard。

### Hard

满足任意情况：

- 核心方向基本正确，但遗漏关键条件
- 依赖关键提示才能完成
- 会背定义，但解释不清原因
- 标准问题能回答，稍微变化就不会
- 面试表达混乱，导致核心信息不完整

#### 内容型 Hard

核心条件遗漏、因果不完整、变式不会、需要关键提示。

处理：后退一个阶段。

#### 表达型 Hard

理解正确、术语不完整、表达不够面试化、没有影响核心结论。

处理：状态不升级，记录表达薄弱点，间隔保持或小幅缩短，不强制后退一级。核心理解完整时，可评为 Good 并将"正式表达不足"单独记录为薄弱点。

### Good

需要满足：

- 无关键提示
- 核心答案正确
- 主要逻辑完整
- 能解释关键原因
- 能完成当前阶段要求的标准应用

专业术语稍有不准确，但核心理解正确时，可以评为 Good，并补充标准表达。

### Easy

需要表现出：

- 无提示
- 准确
- 表达流畅
- 能解释原因
- 能说明边界或对比
- 能处理变式、迁移或项目场景

回答速度快不等于 Easy。

### 选择题特殊规则

选择题选对后，不能立即评为 Good 或 Easy。至少追问：

- 为什么选择这个选项？
- 一个相似选项为什么不成立？

无法解释时，最多评为 Hard。

## 主线与支线分别评分

主知识点与关联支线分别判断、分别评分、分别调度。

支线回答错误，只影响该支线或主线对应的具体薄弱点，不得直接覆盖主知识点已经取得的成绩。

---

## 即时理解确认

微型教学后立即答对，只算"即时理解确认"，不立即产生正式评分。

记录方式：

```markdown
- 不完整类型：
  - 本次状态：首次教学
  - 即时理解：通过
  - 正式评分：待延迟验证
  - 与主线关系：解释 Pimpl 析构函数为什么放在 cpp
```

复习调度表暂时不写 Good，等完成变式或跨日验证后，再产生正式评分。

## 复习间隔阶梯

```
1天 → 3天 → 7天 → 14天 → 30天 → 60天
```

### Good 的处理

Good 后前进一个阶段：

```
1天 → 3天
3天 → 7天
7天 → 14天
14天 → 30天
30天 → 60天
```

### Easy 的处理

Easy 默认前进一个阶段。

强 Easy（跳两级）需同时满足：

- 无提示
- 核心与细节准确
- 完成未见变式或真实应用
- 能说明边界和对比
- 没有待验证薄弱点
- 此前至少有一次跨日 Good

首次学习即使表现很好，也不应从 1 天直接跳到 7 天。

### Hard 的处理

内容型 Hard 后退一个阶段：

| 当前阶段 | Hard 后阶段 | 下次复习 |
|----------|-------------|----------|
| 1天 | 1天 | 1天后 |
| 3天 | 1天 | 1天后 |
| 7天 | 3天 | 3天后 |
| 14天 | 7天 | 7天后 |
| 30天 | 14天 | 14天后 |
| 60天 | 30天 | 30天后 |

Hard 后下一次 Good 只能恢复一级，不能立即跳回更长间隔。

### Again 的处理

```
针对性重新讲解
→ 当天延迟变式验证
→ 次日跨日验证
→ 复习阶段回到1天
```

---

## 针对薄弱点安排下次复习

复习不能只调整日期，还要记录上一次暴露的问题。例如：

- 定义会，场景不会
- 核心机制会，细节不清
- 会说，不会写
- 概念理解正确，但容易与另一个概念混淆
- 标准题会，变式不会

下一次复习优先针对薄弱点，不能随机重问无关内容。

---

## 复习角度逐渐变化

### 1天阶段

检查：核心定义、解决的问题、主要机制、为什么有效。

### 3天阶段

检查：原因、相似概念对比、基础应用、关键条件。

### 7天阶段

检查：边界、重要细节、变式、错误诊断。

### 14天及以后

检查：面试连续追问、真实场景、项目应用、综合设计、实践和迁移。

## 错题闭环

### 保留原始评分

原题答错就是 Again。即使重新讲解后验证通过，也不得将原始 Again 改成 Good。

### 判断错误类型

- 知识缺失
- 因果不清
- 概念混淆
- 不会应用
- 步骤错误
- 依赖提示
- 粗心或口误

### 针对错误重新讲解

不得原样重复之前的讲解。应根据错误换一种方式：

- 原来用文字，改为图示
- 原来用定义，改为案例
- 原来展示正确做法，改为分析错误做法
- 原来整体讲，改为拆分步骤

只补用户真正缺失的部分，不从头重讲已经掌握的全部内容。

### 当天延迟验证

答错后不得立刻重复原题。应先穿插其他题目或知识，再使用变式题验证。

不得为了制造延迟而机械增加低价值题目。如果当前会话没有自然间隔，应将验证标记为待完成。

### 跨日再次验证

当天答错的知识，即使延迟验证通过，也必须在下一次跨日学习中再次无提示考察。跨日通过后，才能恢复正常复习阶段。

## 知识状态判定

### `[ ]` 未学习

从未正式学习。

### `[~]` 学习中

满足任意情况：

- 仍依赖讲解或提示
- 只能复述定义
- 无法解释原因
- 无法完成基础应用
- 当天刚完成错题纠正
- 仍有待验证内容

### `[x]` 已理解

至少满足：

- 能用自己的话解释
- 能说明主要原因
- 能区分关键近邻概念（如存在）
- 能完成与知识类型匹配的基础验证
- 当前没有根本性误解

基础验证按知识类型区分：

- 事实型：准确回忆和正确使用
- 概念型：因果解释与辨析
- 程序型：独立完成步骤
- 策略型：场景选择和取舍

### `[✓]` 已巩固

至少满足：

- 在两个不同日期无提示成功提取
- 完成过变式、辨析或案例应用
- 不是只会原题
- 当前没有未完成的 Again 验证

### `[★]` 已掌握

仅对核心重要知识使用。至少满足：

- 在三个或更多不同日期成功提取
- 能解释原因和边界
- 能承受一轮面试追问
- 能完成未见过的变式
- 能分析错误
- 完成实践或综合迁移
- 不依赖关键提示

不是所有小知识都必须达到 `[★]`。

### 状态降级

- 出现 Again：降为 `[~]`，重新建立理解和跨日验证
- 连续两次 Hard：下降一级
- 单次轻度 Hard：通常保留状态，但记录薄弱点，禁止升级
