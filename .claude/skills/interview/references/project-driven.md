# 项目驱动与探索式学习规则

## 核心定位

当用户提出以下类型的目标时，进入项目驱动模式：

- 想实现一个功能
- 想从 0 到 1 制造一个小玩具
- 想理解自己项目中的一段代码或机制
- 对某个技术现象产生明确好奇
- 想修复项目问题或验证一个猜想

项目驱动模式下，项目目标优先于按照知识点清单顺序教学。

Tracker 继续负责：

- 记录知识缺口
- 安排复习
- 防止遗漏核心知识
- 记录实践证据
- 形成面试表达

---

## 默认执行路线

```
明确想做什么或想弄清什么
→ 查看当前项目或最小场景
→ 找到当前真正阻塞点
→ 判断需要哪些最小知识
→ 对未学知识先教学
→ 制作或修改最小版本
→ 运行与观察
→ 主动制造一个失败场景
→ 修复并增加测试
→ 总结原理、取舍和限制
→ 更新 tracker
```

---

## 硬规则

1. 不得先讲完一个完整理论模块，再允许用户进入项目。
2. 不得只帮用户生成代码而不解释关键设计。
3. 不得把 AI 生成并运行成功视为用户已经掌握。

---

## 产物要求

每次项目驱动学习应尽量产生至少一种证据：

- 可运行的小程序
- 一段用户亲自完成的代码
- 调用链图
- 流程图
- 测试
- Trace
- 实验记录
- Bug 修复
- 设计对比
- 项目改进说明

产物完成后，必须验证用户能否解释：

- 它解决什么问题
- 核心流程
- 为什么这样实现
- 替代方案
- 可能怎样失败
- 如何测试或观察

---

## 项目中出现新知识时

项目问题可以先用于诊断，但不得把未教学内容直接作为正式考试。

流程：

```
项目中发现未知概念
→ 明确它为什么与当前目标有关
→ 最小教学
→ 即时理解确认
→ 回到项目使用
→ 后续再做正式变式或跨日评分
```

---

## 防止只追求新鲜感

每个小玩具至少完成以下闭环：

```
做出来
→ 跑起来
→ 弄坏一次
→ 找到原因
→ 修复
→ 写测试或验证证据
→ 用自己的话解释
→ 写入 tracker
```

未完成闭环前，不应无理由切换到新的玩具。

---

## 知识网络限制

项目驱动模式下，知识网络扫描以当前项目有用为原则：

- 默认只展示 1～3 个高价值连接
- 优先展开当前阻塞所需的连接
- 其他连接登记为后续方向
- 不得因为知识网络发散而长期离开当前项目目标
