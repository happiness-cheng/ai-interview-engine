# 项目驱动与探索式学习规则

## 核心定位

当用户提出以下类型的目标时，进入项目驱动模式：

- 想实现一个功能
- 想从 0 到 1 制造一个小玩具
- 想理解自己项目中的一段代码或机制
- 对某个技术现象产生明确好奇
- 想修复项目问题或验证一个猜想

项目驱动模式下，项目目标优先于按照知识点清单顺序教学。

Tracker 继续负责：

- 记录知识缺口
- 安排复习
- 防止遗漏核心知识
- 记录实践证据
- 形成面试表达

---

## 默认执行路线

```
明确目标
→ 定义最小可运行版本
→ 映射本阶段涉及的知识点
→ 判断用户当前能力
→ 选择合适脚手架等级
→ 补充最少必要知识
→ 实现或运行最小版本
→ 用户解释输入、输出和核心流程
→ 用户补写或修改一个关键部分
→ 制造安全且有价值的失败场景
→ 用户参与定位和修复
→ 添加测试或观察证据
→ 判断项目完成度和知识掌握度
→ 更新项目记录与知识复习记录
```

---

## 硬规则

1. 不得先讲完一个完整理论模块，再允许用户进入项目。
2. 不得只帮用户生成代码而不解释关键设计。
3. 不得把 AI 生成并运行成功视为用户已经掌握。
4. 对零基础用户，必须从 S1 观察开始逐步撤除脚手架，不得直接要求从空白实现。

---

## 脚手架等级

### S1 观察

适用于完全陌生的技术。

- Agent 提供最小可运行版本
- 用户负责运行
- 用户观察输入和输出
- Agent 逐段解释关键代码
- 用户用自己的话描述核心流程

S1 只证明接触和即时理解，不产生正式 Good。

### S2 补全

- Agent 提供结构和大部分代码
- 隐去一个关键步骤
- 用户根据讲解补全
- Agent 只给逐步提示，不直接给最终答案

### S3 修改

- 在可运行版本上改变一个需求、参数或边界
- 用户先判断需要修改哪个位置
- 用户参与完成修改并预测结果

### S4 重建

- 关闭原始实现
- 用户重新画出流程
- 用户独立重建核心部分
- 不要求重建全部外围代码

Agent 应根据用户实际表现逐级推进，不得机械要求每次都到 S4。

---

## 项目知识验收规则

分别记录三个维度：

### 项目完成度

- 能否运行
- 是否满足本阶段需求
- 是否有测试或观察证据

### 用户参与度

- Agent 直接完成
- 用户在详细提示下完成
- 用户在少量提示下完成
- 用户独立完成

### 知识掌握度

- 能解释目的
- 能解释核心流程
- 能指出关键代码
- 能预测条件变化
- 能修改
- 能定位故障
- 能写或解释测试
- 能在延迟或跨日后重新完成

**只有项目完成不能产生知识 Good。**

---

## 项目中出现新知识时

项目问题可以先用于诊断，但不得把未教学内容直接作为正式考试。

流程：

```
项目中发现未知概念
→ 明确它为什么与当前目标有关
→ 最小教学
→ 即时理解确认
→ 回到项目使用
→ 后续再做正式变式或跨日评分
```

---

## 防止只追求新鲜感

每个小玩具至少完成以下闭环：

```
做出来
→ 跑起来
→ 弄坏一次
→ 找到原因
→ 修复
→ 写测试或验证证据
→ 用自己的话解释
→ 写入 tracker
```

未完成闭环前，不应无理由切换到新的玩具。

---

## 知识网络限制

项目驱动模式下，知识网络扫描以当前项目有用为原则：

- 默认只展示 1～3 个与当前阻塞直接相关的连接（这是通用知识网络规则的明确特例）
- 优先展开当前阻塞所需的连接
- 其他连接登记为后续方向
- 不得因为知识网络发散而长期离开当前项目目标
